### Bug #1: 検索フォームで負の価格値が受け入れられる

**重要度**: 🟡 中

**カテゴリ**: 入力バリデーション不備

**再現手順**:
1. ブラウザで `http://127.0.0.1:5000` にアクセス
2. 管理者またはユーザーでログイン
3. 商品一覧ページの検索フォームで「価格（下限）」に `-1` を入力
4. 「検索」ボタンをクリック

**期待される動作**:
- 価格は0円以上であるべきなので、負の値は受け入れるべきではない
- エラーメッセージを表示するか、入力を拒否すべき

**実際の動作**:
- 負の値を入力しても検索が実行される
- SQLクエリに `-1` が渡され、すべての商品が結果として表示される（price >= -1 はすべて真）

**技術的詳細**:
- 検索フォームの `<input type="number">` に `min="0"` 属性が設定されていない
- バックエンドでも負の値のバリデーションが行われていない
- 該当ファイル: `templates/products_list.html` 38-43行目、`app.py` 136-139行目

**影響**:
- ユーザーが無効な検索条件を指定でき、予期しない検索結果により混乱する可能性がある
- セキュリティリスクは低いが、UX（ユーザー体験）の観点で問題

**推奨修正**:
1. HTMLテンプレートの入力フィールドに `min="0"` 属性を追加
2. バックエンドで負の値が渡された場合のバリデーションを追加

```html
<!-- 修正例 -->
<input type="number" 
       id="price_min" 
       name="price_min" 
       data-testid="search-price-min"
       value="{{ request.args.get('price_min', '') }}"
       min="0"
       placeholder="例: 1000">
```

### Bug #2: 検索で price_min > price_max が許可される

**重要度**: 🟡 中

**カテゴリ**: ロジックバリデーション不備

**再現手順**:
1. ブラウザで商品一覧ページにアクセス
2. 検索フォームで：
   - 「価格（下限）」に `100` を入力
   - 「価格（上限）」に `50` を入力
3. 「検索」ボタンをクリック

**期待される動作**:
- 下限が上限より大きい場合、エラーメッセージを表示すべき
- または、自動的に値を入れ替えるべき
- 最低限、ユーザーに警告を表示すべき

**実際の動作**:
- 検索は実行されるが、論理的に不可能な条件（100円以上 AND 50円以下）のため結果は常に0件
- エラーメッセージやワーニングは表示されない
- URL: `http://127.0.0.1:5000/products?keyword=&category=&price_min=100&price_max=50`

**技術的詳細**:
- バックエンドで価格範囲の論理的整合性チェックが実装されていない
- SQLクエリ: `WHERE price >= 100 AND price <= 50` （常に偽）
- 該当ファイル: `app.py` 136-143行目

**影響**:
- ユーザーが意図しない検索結果（常に0件）を得て混乱する
- サポートへの問い合わせ増加の可能性
- ユーザビリティの低下

**推奨修正**:
バックエンドで価格範囲の検証を追加：

```python
# app.py に追加
if price_min and price_max:
    if int(price_min) > int(price_max):
        flash('価格の下限は上限以下である必要があります。', 'error')
        return redirect(url_for('products_list'))
```

### Bug #4: 商品新規登録フォームが送信できない 🔴

**重要度**: 🔴 **高（重大）**

**カテゴリ**: フォーム実装不備、機能障害

**再現手順**:
1. 管理者アカウント（admin / admin_password）でログイン
2. ナビゲーションバーの「新規登録」をクリック
3. 商品登録フォームで以下を入力：
   - 商品名: `テスト商品`
   - カテゴリ: `書籍`
   - 価格: `1000`
   - 在庫数: `10`
   - 商品説明: （任意）
4. 「登録」ボタンをクリック

**期待される動作**:
- 商品が登録される
- 「商品を登録しました。」というメッセージが表示される
- 商品一覧ページにリダイレクトされる

**実際の動作**:
- フォームが送信されない
- 同じページ（商品登録フォーム）に留まる
- エラーメッセージも表示されない
- **商品が一切登録できない**

**根本原因**:
ステータスのラジオボタンに `disabled` 属性が設定されているため、フォーム送信時にその値がサーバーに送信されません。

**該当コード**: `templates/product_form.html` 83-89行目
```html
<input type="radio" 
       id="status_preparing" 
       name="status" 
       value="準備中" 
       checked
       disabled>  <!-- ← この disabled が問題の原因 -->
<label for="status_preparing" style="display: inline; font-weight: normal;">
    準備中（新規登録時は固定）
</label>
```

**技術的説明**:
- HTMLの仕様では、`disabled` 属性が設定された入力要素の値はフォーム送信時に含まれません
- バックエンド（`app.py` 169行目）では `status = '準備中'` とハードコードされていますが、フォーム送信自体が失敗している可能性
- ブラウザの開発者ツールで `disabled` 属性を削除すると、正常に登録できることを確認済み

**影響**:
- 🔴 **クリティカル**: 通常の操作では商品を新規登録できない
- 管理者が商品を追加できないため、システムの基本機能が利用不可
- ビジネスへの直接的な影響が大きい

**回避策**:
ブラウザの開発者ツールを使用して、以下のJavaScriptを実行：
```javascript
document.querySelector('input[name="status"]').disabled = false;
```
その後、フォームを送信すると正常に登録できます。

**推奨修正方法**:

**方法1**: hidden inputを使用（推奨）
```html
<div>
    <!-- 表示用（disabled） -->
    <input type="radio" 
           id="status_preparing_display" 
           value="準備中" 
           checked
           disabled>
    <!-- 送信用（hidden） -->
    <input type="hidden" 
           name="status" 
           value="準備中">
    <label for="status_preparing_display" style="display: inline; font-weight: normal;">
        準備中（新規登録時は固定）
    </label>
</div>
```

**方法2**: readonly スタイルを適用（disabled を削除）
```html
<input type="radio" 
       id="status_preparing" 
       name="status" 
       value="準備中" 
       checked
       onclick="return false;"
       style="pointer-events: none; opacity: 0.6;">
```

**方法3**: バックエンドで強制設定（現在の実装を活かす）
```python
# app.py の product_new 関数
status = '準備中'  # フォームからの値に関係なく常に準備中を設定
```
ただし、この場合でもフロントエンドの disabled は削除すべきです。
